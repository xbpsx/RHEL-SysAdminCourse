**Жёсткая ссылка (hard link)** — это дополнительное имя для уже существующего файла.

**Простая аналогия:** У одного человека может быть несколько документов с разными именами (паспорт, водительские права), но все они указывают на одну и ту же личность.

### Как это работает технически:

1. **Один файл = один inode** с данными на диске.

2. **Каждое имя файла** в каталоге — это ссылка на этот inode.

3. **Жёсткая ссылка** — это просто **ещё одна запись в каталоге**, которая указывает на ТОТ ЖЕ САМЫЙ inode.


**Пример:**

```bash
# Создаём оригинальный файл
echo "Привет" > original.txt

# Создаём жёсткую ссылку на него
ln original.txt hardlink.txt
```

# Теперь оба имени указывают на один inode

---
### Ключевые свойства жёстких ссылок:

✅ **Полная равноправность** — нет понятия "оригинал" и "ссылка", оба имени абсолютно равнозначны  
✅ **Неотличимы друг от друга** — если удалить "оригинал", данные останутся доступны через "ссылку"  
✅ **Занимают минимум места** — создаются мгновенно, не копируют данные  
❌ **Работают только в пределах одной файловой системы**  
❌ **Нельзя создать для директорий** (обычно)

---

### Чем отличается от символической (мягкой) ссылки:

|Жёсткая ссылка|Символическая ссылка|
|---|---|
|Указывает НАПРЯМУЮ на inode|Указывает на ИМЯ файла|
|Работает при удалении "оригинала"|Становится битой при удалении оригинала|
|Только в одной ФС|Может ссылаться между разными ФС|

**На практике:** Жёсткие ссылки полезны для экономии места когда нужно иметь один файл в нескольких местах файловой системы.

---
**Нет, это не копирование!** Это важное отличие. Давайте на примере:

У нас есть файл `text.txt` с содержимым `"Важные данные"`.

### Копирование (`cp text.txt copy.txt`)

```text
text.txt  →  данные "Важные данные" (inode 100)
copy.txt  →  данные "Важные данные" (inode 200) ← НОВЫЕ данные!
```

**Результат:** Два независимых файла, занимают в 2 раза больше места.

---
### Жёсткая ссылка (`ln text.txt hardlink.txt`)

```text
text.txt     →  данные "Важные данные" (inode 100)
hardlink.txt ↗
```

**Результат:** Один набор данных, два равноправных имени. Занимают место как один файл.

---
### Символическая ссылка (`ln -s text.txt symlink.txt`)

```text
text.txt    →  данные "Важные данные" (inode 100)
symlink.txt →  указатель "→ text.txt" (inode 300) ← Ярлык!
```

---
### Краткий итог для вашего случая:

|Ваша цель|Что выбрать|
|---|---|
|**Создать независимую копию** (чтобы изменять отдельно)|`cp`|
|**Сэкономить место** (один файл, несколько имён)|`ln` (жёсткая ссылка)|
|**Создать "ярлык"** (для удобства доступа)|`ln -s` (символическая ссылка)|
|**Перенести в другую файловую систему**|`ln -s` (только симлинк)|

**Для вашего примера:** Если хотите чтобы файл был доступен из другой папки, но при этом экономить место — используйте жёсткую ссылку. Если файловая система другая — только симлинк.

---

### **Ваша ситуация:**

- Файл: `/home/username/.config/config.json` (размер: 2 KB)

- Inode: №1000

### **Зачем вам хардлинк?**

**Представьте сценарий:** У вас есть программа, которая **ищет свой конфиг только в `/etc/myapp/config.json`**, но вы хотите хранить все свои конфиги в домашней директории.

**Решение с хардлинком:**

```bash
# Создаём хардлинк
sudo ln /home/username/.config/config.json /etc/myapp/config.json
```

**Что произошло в системе:**

```text
/home/username/.config/config.json  →  данные (inode 1000)
/etc/myapp/config.json
```

---
### **Как это экономит место - ТЕХНИЧЕСКИ:**

1. **Без хардлинка (копирование):** 

```bash
sudo cp /home/username/.config/config.json /etc/myapp/config.json
```

- На диске создаётся **НОВАЯ копия данных** (ещё 2 KB)

- **Итого:** 4 KB занято + 1 inode

- **С хардлинком:**
    
    - На диске **остаются те же самые 2 KB данных**
    
    - В каталоге `/etc/myapp/` просто появляется **новая запись** "config.json → inode 1000"
    
    - **Итого:** 2 KB занято (экономия 50%) + 0 новых inode

```bash
ls -li /home/username/.config/config.json /etc/myapp/config.json
# Оба файла покажут ОДИННАКОВЫЙ номер inode
```

### **Что даёт на практике:**

✅ **Синхронизация:** Изменение в одном файле мгновенно видно в другом (это один файл!)  
✅ **Экономия места:** В 2 раза меньше занимаете места на диске  
✅ **Централизованное управление:** Правите конфиг в одном месте, изменения применяются везде

### **Когда НЕЛЬЗЯ использовать хардлинк:**

❌ Если нужно перенести конфиг на **другой диск** или в **другую файловую систему**  
❌ Если вы хотите, чтобы одна программа могла изменять конфиг, а другая видела старую версию

**Для этих случаев используйте симлинк:**
```bash
sudo ln -s /home/username/.config/config.json /etc/myapp/config.json
```

**Вывод:** Хардлинк — это не "копия", а "второе паспортное имя" для того же самого файла. Экономия идёт за счёт того, что данные на диске не дублируются.
